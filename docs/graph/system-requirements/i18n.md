# SR-8: Internationalization (i18n)

**Functional Requirements**:
[FR-8.1, FR-8.2](../functional-requirements.md#fr-8-internationalization-i18n)

## User Stories

- As a non-English speaker, I want to use the application in my native language so I can understand
  the interface and work effectively.
- As a user, I want the application to automatically display in my browser's language preference so
  I don't have to manually switch languages.

## Acceptance Criteria

- [SR-8.1](#sr-81-application-detects-browser-language-preference-on-first-load): Application
  detects browser language preference on first load
- [SR-8.2](#sr-82-ui-strings-are-fully-localized--no-interface-text-remains-in-a-fixed-language-when-a-supported-locale-is-active):
  UI strings are fully localized — no interface text remains in a fixed language when a supported
  locale is active
- [SR-8.3](#sr-83-translations-fall-back-to-english-if-a-string-is-missing-for-a-locale):
  Translations fall back to English if a string is missing for a locale

## Scenarios

### SR-8.1: Application detects browser language preference on first load

#### Preconditions

- User opens Architeezy Graph for the first time (or with cleared storage)
- The user's browser is configured with a non-English language preference
- That language is among the supported locales

#### Steps

1. **Open the application**
   - The application reads the browser's language preference on startup
   - The best-matching supported locale is selected (exact match, then prefix match, then English)

2. **Observe the interface language**
   - All UI elements (buttons, labels, menus, error messages) are displayed in the detected language
   - No manual language selection is required

3. **Verify model content is unaffected**
   - Element names and other content from the graph model are displayed in their original language
   - Only interface strings are localized

#### Edge Cases

- **User's browser language not supported** — the application falls back to English
- **Translation file fails to load** — the application falls back to English translations and shows
  an error notification; the user can continue in English

### SR-8.2: UI strings are fully localized — no interface text remains in a fixed language when a supported locale is active

#### Preconditions

- The application has started with a supported non-English locale active
- The translation bundle for that locale has loaded successfully

#### Steps

1. **Inspect buttons and controls throughout the UI**
   - All button labels, tooltips, and control text appear in the active locale's language

2. **Inspect menus, panels, and dialogs**
   - All menu items, panel headings, and dialog text appear in the active locale's language
   - No hardcoded English strings are visible in the interface

3. **Trigger error and notification messages**
   - Error messages, toast notifications, and help text are displayed in the active locale's
     language

#### Edge Cases

- **Partial translation (some keys exist, some missing)** — existing keys are shown in the selected
  language; missing keys fall back to English; no errors or broken UI occur

### SR-8.3: Translations fall back to English if a string is missing for a locale

#### Preconditions

- A supported non-English locale is active
- At least one translation key is absent from the active locale's bundle

#### Steps

1. **Navigate to a UI area with a missing translation key**
   - The string for the missing key is displayed in English
   - The rest of the interface remains in the active locale's language

2. **Verify no broken UI**
   - No raw translation key identifiers are visible to the user
   - The UI layout is unaffected by the missing translation

3. **Observe fallback chain**
   - For a region-specific locale (e.g., `fr-FR`), the system first tries the base locale (`fr`)
     before falling back to English

#### Edge Cases

- **Corrupted translation bundle** — the application detects the error, falls back to the English
  bundle, and logs the issue; the UI remains functional
- **Translation key missing in bundled translations** — the English fallback string is displayed; a
  warning is logged in development for translation teams; no visible key identifier appears in the
  UI

## Business Rules

### Locale Determination Priority

1. Detect from browser's language preferences (in priority order)
2. First supported locale in the list → use it
3. Default to English if none supported

### Translation Bundle Structure

- Bundles are JSON files organized by locale (e.g., `locales/en.json`, `locales/fr.json`)
- Filename uses standard language tags (e.g., `en`, `fr`, `pt-BR`)
- For region-specific locales like `fr-FR`, support fallback to base locale `fr`
- JSON structure: flat key-value pairs, keys dot-delimited for namespacing
- Translations may contain placeholders for dynamic values using template syntax (e.g., `{{count}}`)

### Translation Function

A central translation function looks up keys in the current locale's bundle, falls back to English
if missing, and applies placeholder replacements with dynamic values.

## Technical Notes

### Translation Bundle Loading

Use dynamic import for locale-specific JSON bundles if the build system supports it, or load via
HTTP request. Support fallback to base locale and English if specific locale file is not found.

### Application State

Application state includes: current locale, list of available locales, and loaded translation
bundles.

### Integration with Existing Code

Search codebase for hardcoded user-facing strings and replace with translation function calls.
Define translation keys using a consistent naming convention. Never translate data coming from the
model.

### Testing

Verify translations for each supported locale, test interpolation scenarios, and check fallback
chain behavior.
