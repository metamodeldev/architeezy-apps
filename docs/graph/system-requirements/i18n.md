# SR-8: Internationalization (i18n)

**Functional Requirements**:
[FR-8.1, FR-8.2](../functional-requirements.md#fr-8-internationalization)

## User Stories

- As a non-English speaker, I want to use the application in my native language so I can understand
  the interface and work effectively.
- As a user, I want the application to automatically display in my browser's language preference so
  I don't have to manually switch languages.

## Acceptance Criteria

- SR-8.1: Application detects browser language preference on first load
- SR-8.2: UI strings are fully localized — no interface text remains in a fixed language when a
  supported locale is active
- SR-8.3: Translations fall back to English if a string is missing for a locale

## Scenario

### Preconditions

- User opens Architeezy Graph for the first time (or with cleared storage)
- Supported languages: English (default), plus any additional translations available

### Steps

1. **Initial Language Detection**
   - On app startup, read browser's language preference
   - Compare against available translation locales
   - Select best match: exact match → prefix match → fallback to English
   - Store detected/selected locale in memory

2. **Load Translation Strings**
   - Load the appropriate translation bundle for the selected locale
   - Bundle contains key-value pairs for UI strings
   - If translation file fails to load or locale not supported, fall back to English
   - Current locale is set in application state

3. **Render UI with Translations**
   - All UI components use a translation function instead of hardcoded strings
   - On render, component calls the translation function with a string key
   - Missing keys log warning in development and fall back to the key itself or English
   - Model data (element names from the graph) are passed through without translation

4. **Translation Fallback**
   - For any translation key missing in the selected locale, the system falls back to English
   - Missing keys are logged as warnings during development for translation teams
   - The fallback chain follows: current locale → base locale → English

### Expected Results

- Application starts in the user's browser language if available, otherwise English
- All buttons, menus, labels, error messages, and help text are properly localized
- Element names and model content remain in their original language
- Missing translations gracefully fall back to English without broken UI

### Edge Cases

- **User's browser language not supported**
  - Falls back to English

- **Translation file fails to load (network error)**
  - Fall back to English translations
  - Show error notification; user can retry or continue in English

- **Corrupted translation JSON**
  - Detect parsing error
  - Fall back to English bundle
  - Clear invalid data and log error

- **Translation key missing in bundled translations**
  - Display key or fallback string (English)
  - Log warning in console for translation teams
  - UI remains functional (no visible key if fallback configured)

- **Partial translation (some keys exist, some missing)**
  - Existing keys are shown in selected language
  - Missing keys fall back to English
  - No jank or errors in UI

## Business Rules

### Locale Determination Priority

1. Detect from browser's language preferences (in priority order)
2. First supported locale in the list → use it
3. Default to English if none supported

### Translation Bundle Structure

- Bundles are JSON files organized by locale (e.g., `locales/en.json`, `locales/de.json`)
- Filename uses standard language tags (e.g., `en`, `de`, `pt-BR`)
- For region-specific locales like `de-DE`, support fallback to base locale `de`
- JSON structure: flat key-value pairs, keys dot-delimited for namespacing
- Translations may contain placeholders for dynamic values using template syntax (e.g., `{{count}}`)

### Translation Function

A central translation function looks up keys in the current locale's bundle, falls back to English
if missing, and applies placeholder replacements with dynamic values.

## Technical Notes

### Translation Bundle Loading

Use dynamic import for locale-specific JSON bundles if the build system supports it, or load via
HTTP request. Support fallback to base locale and English if specific locale file is not found.

### Application State

Application state includes: current locale, list of available locales, and loaded translation
bundles.

### Integration with Existing Code

Search codebase for hardcoded user-facing strings and replace with translation function calls.
Define translation keys using a consistent naming convention. Never translate data coming from the
model.

### Testing

Verify translations for each supported locale, test interpolation scenarios, and check fallback
chain behavior.
